Help & explanations
Overview

This page shows automatic background jobs that run inside the application process while the server is up. You do not need to keep this page open — tasks keep running in the background.

The main job, fetch_weather, wakes up every 1 minute and checks which cities are due for a weather update. A city is due when it has never been checked, or when last_checked_at is older than its effective check interval (per-city setting, or the global default from Settings). Only due cities are fetched; others are skipped until their interval expires.

Manual alternatives: Tools → Fetch due cities, Fetch now on a city details page, or the CLI command flask fetch-weather.

{% if not scheduler_enabled %}
Scheduler disabled

The in-process APScheduler was not started for this web instance. In Docker deployments, automatic fetching usually runs in a separate worker container via flask run-worker.

To use the in-process scheduler (for example during local development), set INTERNAL_SCHEDULER_ENABLED=true and restart the server. For production-style deployments, keep INTERNAL_SCHEDULER_ENABLED=false on web and run flask run-worker in a dedicated process or container.

Column guide
ID
Internal identifier used by the scheduler and action buttons. The default weather job is always fetch_weather.
Name
Human-readable label from APScheduler. May be empty for jobs added without an explicit name.
Trigger
When and how the job repeats. For example, interval[0:01:00] means “run every 1 minute”.
Next Run Time
The next scheduled start time while the job is active. A dash (-) means the job is paused and nothing is scheduled. This is not the time of the last run.
Status
Current scheduling state (see below). Status is evaluated when this page loads; refresh the page to update it.
Status badges

Status describes the schedule, not whether the whole application is busy. A job can be Active most of the time even though each run only takes a moment.

  • Active — The job is enabled and waiting for its next scheduled run. fetch_weather will keep checking for due cities on schedule.
  • Executing — The job is running right now (for example, calling weather APIs for due cities). This state is usually brief; reload the page if you want to see it.
  • Paused — The job is still registered but will not run on schedule. No new automatic weather fetches happen until you click Resume.
Action buttons
Temporarily stop future scheduled runs. The job stays in the list and can be turned back on with Resume. If a run is already in progress, it is not interrupted — only the next scheduled start is prevented.
Re-enable a paused job. The scheduler calculates a new Next Run Time and automatic fetching continues.
Permanently remove the job from the scheduler in this process. Despite the label, this is not “cancel the current run” — it deletes the scheduled task entirely. Automatic weather fetching stops until the application is restarted, which recreates the default fetch_weather job. Use Pause if you only need a temporary break.
About fetch_weather

On each run, fetch_weather calls fetch_due_cities(), which:

  1. Loads the global default check interval from Settings.
  2. Iterates over every city in the database.
  3. For each city, compares last_checked_at with that city's interval (or the default if the city interval is empty).
  4. Fetches fresh weather only for cities that are due; skips the rest for this run.
  5. On API or geocoding errors for a single city, logs the failure and continues with the remaining cities.

Because the scheduler wakes every minute but cities may have longer intervals (for example, 30 or 60 minutes), most runs finish quickly with zero or few API calls. That is expected behavior.
